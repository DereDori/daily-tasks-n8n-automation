Daily TasksAutomation

An n8n workflow that rebuilds your Obsidian daily task file every morning — pulling from calendar events, chore schedules, and yesterday's unfinished work.

// 01 — Overview

What is this?

Every morning at 5 AM, this workflow reads your Obsidian vault stored in Dropbox, processes seven distinct sections according to custom rules, fetches today's calendar events, calculates which household chores are due, and writes a fresh Daily Tasks file — ready before you wake up.

It also archives yesterday's original file to a dedicated archive folder, preserving a full history of every day's focus.

// 04 — Functionality

Section Processing Rules

Each of the 7 markdown sections is handled by a distinct rule. This is the core logic of the workflow.

# To do from yesterday's
Rollup
Aggregates all items from yesterday's own section plus the calendar, chores & duties, and randoms sections. All checked items [x] are unchecked to [ ]. Only incomplete items carry forward. The header date link is updated to [[YYYY-MM-DD Day Focus]].
## To do from calander
Replace
Entirely replaced with today's matched calendar events formatted as - [ ] [[Filename without .md]]. If no events match today, the section is left empty.
## To do chores & duties
Replace
Entirely replaced with today's due chores from chores.json. A chore is due if lastDone + (repetition × 86400000ms) ≤ todayEpoch. If lastDone is null the chore is always due.
## To do from Projects
Exact Copy
Copied completely as-is. All nesting, indentation (tabs), checkbox states, and wikilinks are preserved without any modification.
## To do from resolution
Force Reset
Copied but every checkbox is force-reset to [ ] regardless of prior state. Resolutions reappear fresh every single day.
## To do from randoms
Empty
Completely cleared. All previous random items are discarded. The section header is kept but the body is empty.
## To keep in mind
Exact Copy
Copied completely as-is. Persistent notes and reminders that should always be visible.
// 05 — Implementation

Chore Due-Date Logic

Chores are tracked in a flat JSON file with each entry storing a repetition interval in days and the last completed date as an Epoch millisecond timestamp.

// chores.json structure
{
  "chores": [
    { "id": "1", "repetition": 1,  "chorename": "quick Reset",  "lastDone": 1771761600000 },
    { "id": "3", "repetition": 2,  "chorename": "do Dishes",    "lastDone": null },
    { "id": "7", "repetition": 7,  "chorename": "wipe Surfaces", "lastDone": 1771761600000 },
    { "id": "17", "repetition": 90, "chorename": "clean Windows", "lastDone": 1771761600000 }
  ]
}

Due Date Calculation

01
If lastDone is null → chore is always due. It has never been completed.
02
Otherwise: dueEpoch = lastDone + (repetition × 86400000)
03
If dueEpoch ≤ todayEpoch (today at midnight) → chore is due today.
04
Due chores get lastDone updated to todayEpoch − 43200000 (midnight minus 12 hours) and are written back to Dropbox.
05
Non-due chores are passed through unchanged — their lastDone is never touched.

// 07 — Closing Thoughts

Reflections

Building this workflow piece by piece revealed a lot about how n8n handles binary data in filesystem mode — the biggest surprise was that this.helpers.getBinaryDataBuffer is sandboxed out in newer self-hosted versions, requiring the Extract from File node as a workaround.

The Merge + Append pattern for passing context through branching paths turned out to be the most reliable approach. Using items.find() to locate the right item by field presence made the Rebuild node resilient to merge order — a small but important design decision.

The chore system using Epoch milliseconds is deliberately simple — no timezone complexity, no date libraries, just arithmetic. The 12-hour offset on lastDone ensures a chore completed "today" won't resurface again tomorrow even if the repetition interval is 1 day.

The workflow is intentionally modular. Each node does one thing. That made debugging straightforward and will make future changes — like adding a shopping list to chores, or a new section type — easy to slot in without touching the rest of the chain.
